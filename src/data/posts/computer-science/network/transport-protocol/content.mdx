## IP 주소 기반 통신

### ARP(Address Resolution Protocol)

ARP는 IP 주소를 물리적인 주소인 MAC 주소로 변환하는 프로토콜입니다. [IEFT의 RFC 826](https://datatracker.ietf.org/doc/html/rfc826)에서 ARP는 `An Ethernet Address Resolution Protocol` 또는 `Converting Network Protocol Addresses`라고 표현되어 있습니다. 문서에서 따르면 점점 더 많은 네트워크 장비들이 Ethernet을 사용하려 했기에, 다양한 프로토콜 주소를 Ethernet 주소로 변환하는 표준을 제시하려는 목적에서 ARP가 개발되었습니다.

Ethernet은 데이터 링크 계층에서 48비트 하드웨어 주소인 MAC 주소를 사용해서 프레임 전송을 통해 컴퓨터 간에 통신합니다. 이더넷 프레임에는 출발지 MAC 주소와 목적지 MAC 주소가 포함되어 있습니다. 하지만 IP 통신은 네트워크 계층에서 서로 간의 IP 주소만 알 수 있습니다. 계층 간의 주소 체계의 차이를 해결하는 것이 ARP입니다.

### ARP(Address Resolution Protocol) 시나리오

호스트 A (IP: 13.13.10.1)가 호스트 C (IP: 13.13.10.3)로 ICMP Echo 요청을 보내려는 상황을 가정합니다. 여기서 호스트 A는 C의 IP 주소는 알고 있지만 MAC 주소는 알지 못하는 상태입니다.

#### 1. 호스트 A (13.13.10.1) -> 호스트 C (13.13.10.3) ICMP Echo 요청

| 프로토콜 | 출발지 (Source)     | 목적지 (Destination) |
| -------- | ------------------- | -------------------- |
| ICMP     | IP: 13.13.10.1      | IP: 13.13.10.3       |
| IP       | IP: 13.13.10.1      | IP: 13.13.10.3       |
| ETH      | MAC: 00D0.584C.1278 | MAC: X (알 수 없음)  |

- Ethernet 프레임을 작성하기 위해서는 상대방의 MAC 주소가 필요하지만, 현재 MAC 주소는 "알 수 없음" 상태입니다.

```bash
PC>arp -a
No ARP Entries Found
```

- 이때, A의 네트워크 인터페이스는 먼저 ARP 테이블을 확인하여 목적지의 MAC 주소를 찾으려고 시도합니다. 하지만 아직 C에 대한 ARP 엔트리가 없으므로 다음과 같은 메시지가 나타날 수 있습니다:

#### 2. ARP 요청:

| 프로토콜 | 출발지 (Source)     | 목적지 (Destination) |
| -------- | ------------------- | -------------------- |
| ARP      | MAC: 00D0.584C.1278 | MAC: FFFF.FFFF.FFFF  |
| ETH      | MAC: 00D0.584C.1278 | MAC: FFFF.FFFF.FFFF  |

- 호스트 A는 C의 MAC 주소를 알기 위해 ARP 요청을 생성합니다.
- ARP 요청은 네트워크 내 모든 장치에게 보내는 브로드캐스트 패킷입니다.
- 브로드캐스트는 송신 호스트가 전송한 데이터가 네트워크에 연결된 모든 호스트에게 전송되는 방식입니다.
- 여기서 **MAC: FFFF.FFFF.FFFF**는 이더넷 레벨의 브로드캐스트 주소로, 네트워크 상의 모든 장치가 이 패킷을 수신하게 됩니다.
- 이 브로드캐스트의 목적은 해당 IP 주소(13.13.10.3)를 가진 장치가 자신임을 밝히고 자신의 MAC 주소를 A에게 알리는 것입니다.

#### 3. ARP 응답:

| 프로토콜 | 출발지 (Source)     | 목적지 (Destination) |
| -------- | ------------------- | -------------------- |
| ARP      | MAC: 0010.115C.81E2 | MAC: 00D0.584C.1278  |
| ETH      | MAC: 0010.115C.81E2 | MAC: 00D0.584C.1278  |

- 네트워크 상의 모든 장치는 이 ARP 요청을 수신합니다.
- 그 중 IP 주소가 13.13.10.3인 장치(호스트 C)는 자신의 MAC 주소가 0010.115C.81E2임을 A에게 알리기 위해 ARP 응답을 보냅니다.
- 이 ARP 응답은 브로드캐스트가 아닌 유니캐스트 방식으로 A에게만 전송됩니다.
- 유니캐스트는 고유 주소로 식별된 하나의 네트워크 목적지에 1:1로 데이터를 전송하는 방식입니다.
- 이제 호스트 A는 호스트 C의 IP 주소 13.13.10.3에 해당하는 MAC 주소가 0010.115C.81E2임을 알게 되었습니다.

#### 4. ARP 테이블 업데이트:

```bash
13.13.10.3 0010.115C.81E2
```

- ARP 응답을 받은 A는 자신의 ARP 테이블을 업데이트하여 C의 IP와 MAC 주소를 저장합니다.

#### 5. ICMP Echo 요청 (MAC 주소 확인 후):

| 프로토콜 | 출발지 (Source)     | 목적지 (Destination) |
| -------- | ------------------- | -------------------- |
| ICMP     | IP: 13.13.10.1      | IP: 13.13.10.3       |
| IP       | IP: 13.13.10.1      | IP: 13.13.10.3       |
| ETH      | MAC: 00D0.584C.1278 | MAC: 0010.115C.81E2  |

- 이제 A는 C의 MAC 주소를 알기 때문에, Ethernet 프레임을 완성하여 ICMP Echo 요청을 다시 보냅니다.
- 이제 호스트 A는 호스트 C에게 MAC 주소를 포함한 Ethernet 프레임을 전송하고, C는 이를 받아 ICMP Echo 요청을 처리할 수 있게 됩니다.

## 홉 바이 홉 통신

홉 바이 홉 통신(hop-by-hop communication)은 네트워크 상에서 데이터 패킷이 출발지에서 목적지로 전달되는 과정에서, 각 네트워크 장치(라우터 등)를 통해 단계적으로 전달되는 방식을 의미합니다. 이 방식은 IP 통신의 중요한 특성으로, 패킷이 네트워크의 여러 중간 장치를 거치며 최종 목적지에 도달할 때까지 각 홉에서의 처리 과정을 다룹니다.

IP 통신에서는 각 홉마다 해당 패킷을 전달하는 장치가 [다음 홉](https://www.cisco.com/c/en/us/support/docs/dial-access/floating-static-route/118263-technote-nexthop-00.html?dtid=osscdc000283)을 결정하게 됩니다. 이 과정을 네트워크 레벨에서 설명해 보겠습니다.

### 1. 패킷의 생성과 전송

호스트 A에서 IP 주소 192.168.1.10을 가진 호스트 C로 데이터를 전송한다고 가정합니다. 이 경우, 호스트 A는 먼저 C의 IP 주소를 가지고 패킷을 생성합니다.

이 때, 호스트 A는 자신의 라우팅 테이블을 확인합니다. 라우팅 테이블(routing table)에는 게이트웨이와 모든 목적지에 대해 해당 목적지에 도달하기 위해 거쳐야 할 다음 라우터의 정보를 가지고 있습니다. 게이트웨이(gateway)는 호스트가 다른 네트워크로 패킷을 전송할 때 사용하는 라우터의 IP 주소입니다.

- 목적지 IP (192.168.1.10)가 자신의 네트워크 범위 밖이라는 것을 확인합니다.
- 라우팅 테이블에서 기본 게이트웨이 주소를 찾습니다 (예: 10.0.0.254).

호스트 A는 패킷을 기본 게이트웨이(라우터 1)로 보냅니다. 이 과정에서 호스트 A는 IP 헤더의 목적지를 192.168.1.10으로 설정하지만, 이더넷 프레임의 MAC 주소는 게이트웨이의 MAC 주소로 설정합니다.

### 2. 첫 번째 홉: 라우터 1

호스트 A는 자신의 네트워크 게이트웨이(라우터 1)로 패킷을 보냅니다. 이 때, 호스트 A는 해당 패킷의 목적지가 192.168.1.10임을 IP 헤더에 명시하지만, 이더넷 프레임의 MAC 주소는 자신의 네트워크 내 첫 번째 라우터(게이트웨이)로 설정합니다.

| **프로토콜** | **출발지 (Source)** | **목적지 (Destination)** |
| ------------ | ------------------- | ------------------------ |
| **IP**       | IP: 10.0.0.1        | IP: 192.168.1.10         |
| **ETH**      | MAC: A의 MAC        | MAC: 라우터 1의 MAC      |

라우터 1은 이더넷 프레임을 수신하고, 패킷의 IP 헤더를 확인한 후, 다음 홉으로 어디로 보낼지 결정합니다.

### 3. 다음 홉으로 전달: 라우터 2

라우터 1은 경로를 계산한 후 패킷을 라우터 2로 전달합니다. 이 과정에서 라우터 1은 패킷의 이더넷 헤더를 갱신하여 자신(라우터 1)의 MAC 주소를 출발지로, 라우터 2의 MAC 주소를 목적지로 설정합니다. 그러나 IP 헤더는 변하지 않고 원래의 목적지 IP인 192.168.1.10을 유지합니다.

| **프로토콜** | **출발지 (Source)** | **목적지 (Destination)** |
| ------------ | ------------------- | ------------------------ |
| **IP**       | IP: 10.0.0.1        | IP: 192.168.1.10         |
| **ETH**      | MAC: 라우터 1의 MAC | MAC: 라우터 2의 MAC      |

### 4. 마지막 홉: 목적지 호스트 C로 전송

마지막으로 라우터 2는 목적지 호스트 C로 패킷을 전달합니다. 이 경우에도 마찬가지로 IP 주소는 그대로 유지되지만, 이더넷 프레임의 MAC 주소는 라우터 2에서 호스트 C로 변경됩니다.

| **프로토콜** | **출발지 (Source)** | **목적지 (Destination)** |
| ------------ | ------------------- | ------------------------ |
| **IP**       | IP: 10.0.0.1        | IP: 192.168.1.10         |
| **ETH**      | MAC: 라우터 2의 MAC | MAC: C의 MAC             |

### 5. 홉 바이 홉 통신의 의의

홉 바이 홉 통신은 각 라우터가 자신이 처리할 수 있는 구간(홉)에 대한 정보만 가지고 있고, 다음 홉까지의 전송만 책임지는 방식입니다. 각 홉에서 패킷을 처리한 후에 다음 장치로 전달하기 때문에, 경로 선택, 패킷 분할 및 재조립, 오류 검사와 같은 작업이 각 홉에서 이루어질 수 있습니다.

즉, 출발지에서 목적지까지 패킷이 직접적으로 가지 않고, 중간 장치들을 통해 전송되며, 각 장치는 자신의 역할을 수행하고 다음 장치로 데이터를 넘기는 역할을 담당합니다.
