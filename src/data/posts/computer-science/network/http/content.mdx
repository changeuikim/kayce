## 월드 와이드 웹의 탄생

HTTP(HyperText Transfer Protocol)는 월드 와이드 웹의 근간이 되는 프로토콜입니다. 1989년부터 1991년 사이에 Tim Berners-Lee와 그의 팀에 의해 개발된 HTTP는 단순성을 유지하면서도 유연성을 갖추도록 여러 변화를 거쳐왔습니다.

1989년, CERN에서 일하던 Tim Berners-Lee는 인터넷 상에서 하이퍼텍스트 시스템을 구축하는 제안서를 작성했습니다. 초기에는 'Mesh'라고 불렸던 이 시스템은 1990년 구현 과정에서 '월드 와이드 웹'으로 이름이 바뀌었습니다. 이 시스템은 다음 네 가지 구성 요소로 이루어졌습니다:

1. 하이퍼텍스트 문서를 표현하기 위한 텍스트 형식: HTML (HyperText Markup Language)
2. 이 문서들을 교환하기 위한 간단한 프로토콜: HTTP (HyperText Transfer Protocol)
3. 이 문서들을 표시(및 편집)하기 위한 클라이언트: 최초의 웹 브라우저인 WorldWideWeb
4. 문서에 접근할 수 있게 해주는 서버: httpd의 초기 버전

[HTTP 프로토콜 발전의 역사](https://developer.mozilla.org/en-US/docs/Web/HTTP/Evolution_of_HTTP)는 MDN Web Docs에 상세하게 설명되어 있습니다. 이 글에서는 각 버전별 주요 특징을 살펴보겠습니다.

## HTTP/0.9 - 한 줄의 프로토콜

HTTP의 초기 버전은 매우 단순했습니다. 나중에 HTTP/0.9라고 불리게 된 이 버전은 '한 줄의 프로토콜'이라고도 불립니다. 요청은 단 한 줄로 구성되어 있었고, GET 메소드와 리소스의 경로만을 포함했습니다. 서버에 연결되면 프로토콜, 서버 및 포트가 필요하지 않으므로 전체 URL은 포함되지 않았습니다.

예시:
GET /my-page.html

응답 역시 매우 단순해서, 파일 내용만을 포함했습니다.

## HTTP/1.0 - 확장성의 기반

HTTP/0.9의 한계를 극복하기 위해, HTTP/1.0에서는 다음과 같은 개선이 이루어졌습니다.

- 한 연결당 하나의 요청만을 처리하도록 설계되었습니다.
- 버전 정보가 요청에 포함되기 시작했습니다.
- 응답의 시작 부분에 상태 코드가 추가되었습니다.
- HTTP 헤더의 개념이 도입되어 메타데이터를 전송할 수 있게 되었습니다.
- Content-Type 헤더 덕분에 HTML 파일 외의 다른 문서도 전송할 수 있게 되었습니다.

### 단점 - RRT(Round-Trip Time) 문제

HTTP/1.0은 한 번에 하나의 요청만을 처리할 수 있었습니다. 이는 여러 요청을 보낼 때마다 새로운 3-way handshake를 열어야 했기 때문에 RRT(Round-Trip Time) 증가 이슈가 있었습니다. RTT는 패킷이 목적지에 도달하고 나서 다시 출발지로 돌아오기 까지 걸리는 시간을 말합니다.

## [HTTP/1.1](https://datatracker.ietf.org/doc/html/rfc1945) - 표준화된 프로토콜

HTTP/1.1은 사실 HTTP/1.0이 발표된 이후 불과 몇 달 후에 발표되었습니다.

- 한 번 TCP 초기화를 한 이후에 keep-alive라는 옵션으로 여러 개의 파일을 송수신 할 수 있게 바뀌었습니다.
- 연결 재사용이 가능해져 시간을 절약할 수 있게 되었습니다.
- 파이프라이닝이 추가되어 첫 번째 요청에 대한 응답이 완전히 전송되기 전에 두 번째 요청을 보낼 수 있게 되었습니다.
- 청크 응답을 지원하게 되었습니다.
- 캐시 제어 메커니즘이 도입되었습니다.
- 언어, 인코딩, 타입 등의 컨텐츠 협상이 가능해졌습니다.
- Host 헤더 덕분에 동일한 IP 주소에서 다른 도메인을 호스팅할 수 있게 되었습니다.

## HTTP/2 - 성능 향상을 위한 프로토콜

웹 페이지가 점점 복잡해지면서 HTTP/1.1의 한계가 드러나기 시작했습니다. 이를 해결하기 위해 2015년 5월에 HTTP/2가 공식적으로 표준화되었습니다. 주요 특징은 다음과 같습니다:

- 바이너리 프로토콜로 변경되어 최적화가 용이해졌습니다.
- 멀티플렉싱을 지원하여 하나의 연결로 병렬 요청이 가능해졌습니다.
- 헤더 압축을 통해 중복 데이터 전송을 줄였습니다.
- 서버 푸시 기능을 통해 서버가 클라이언트 캐시에 데이터를 미리 넣을 수 있게 되었습니다.

## HTTP/3 - QUIC 위의 HTTP

HTTP의 다음 주요 버전인 HTTP/3는 전송 계층으로 TCP 대신 QUIC을 사용합니다. QUIC은 UDP 위에서 동작하며, 각 스트림에 대해 독립적으로 패킷 손실 감지와 재전송을 처리합니다. 이를 통해 HTTP/2에 비해 훨씬 낮은 지연 시간을 제공합니다.
