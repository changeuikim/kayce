## 데이터베이스란?

> 데이터베이스(Database)는 일정한 규칙, 혹은 규약을 통해 구조화되어 저장되는 데이터의 모음입니다.

데이터베이스를 제어, 관리하는 통합 시스템인 DBMS(Database Management System)를 종종 DB라고 부르기도 합니다. 데이터베이스 안에 있는 데이터들은 특정 DBMS마다 정의된 쿼리 언어(Query Language)를 통해 삽입, 삭제, 수정, 조회 등을 수행할 수 있습니다.

웹 애플리케이션 등은 DBMS를 통해 데이터베이스에 접근하여 실시간으로 데이터를 읽고 쓰는 작업을 수행합니다.

### 엔티티(Entity)

> 엔티티는 데이터베이스에서 관리하고자 하는 대상을 의미합니다. 예를 들어, 학생, 교수, 강의 등이 엔티티가 될 수 있습니다.

엔티티의 속성(Attribute)은 엔티티가 가지고 있는 특성을 의미합니다. 예를 들어, 학생 엔티티의 속성은 학번, 이름, 학과 등이 될 수 있습니다.

스키마(Schema)는 데이터베이스의 구조를 정의하는 것으로, 각 엔티티의 릴레이션, 속성 등을 정의합니다.

### 릴레이션(Relation)

> 릴레이션은 데이터베이스에서 정보를 구분하여 저장하는 기본 단위입니다. 릴레이션은 튜플(Tuple)과 속성(Attribute)으로 구성되어 있습니다.

튜플은 릴레이션의 각 행을 의미하며, 속성은 릴레이션의 각 열을 의미합니다.

릴레이션은 관계형 데이터베이스에서는 테이블(Table)로 표현됩니다. 예를 들어, MySQL에서는 데이터베이스 안에 여러 개의 테이블이 존재하며, 각 테이블은 레코드(Record)로 구성되어 있습니다.

릴레이션은 NoSQL 데이터베이스에서는 컬렉션(Collection)로 표현됩니다. 예를 들어, MongoDB 에서는 데이터베이스 안에 여러 개의 컬렉션이 존재하며, 각 컬렉션은 문서(Document)로 구성되어 있습니다.

### 속성(Attribute)

> 속성은 릴레이션에서 관리하는 구체적이며 고유한 이름을 갖는 데이터의 단위입니다.

속성은 엔티티의 특성을 나타내며, 속성의 값은 도메인(Domain)에 속해야 합니다.

### 도메인(Domain)

> 도메인은 속성이 가질 수 있는 값의 범위를 의미합니다.

예를 들어, 학생 엔티티의 학과 속성은 '컴퓨터공학과', '전자공학과', '경영학과' 등의 값을 가질 수 있습니다.

### 필드(Field)와 레코드(Record)

> 필드는 레코드를 구성하는 각각의 항목을 의미합니다. 레코드는 필드의 집합으로 이루어진 데이터를 의미합니다.

학생 엔티티의 필드는 학번, 이름, 학과 등이 되며, 레코드는 각 학생의 정보를 의미합니다.

#### 필드(Field)와 속성(Attribute)의 차이점

- 필드(Field): 레코드를 구성하는 각각의 항목
- 속성(Attribute): 릴레이션에서 관리하는 구체적이며 고유한 이름을 갖는 데이터의 단위

### 필드 타입(Field Type)

필드 타입은 필드가 가질 수 있는 데이터의 형식을 의미합니다. 예를 들어, 문자열, 숫자, 날짜 등이 필드 타입이 될 수 있습니다.

| --------- | -------------- | ---------- | -------------------------- | ------------------------- | ---------------- | -------------------------- |

| SMALLINT  | 작은 정수      | 2          | -32,768                    | 32,767                    | 0                | 65,535                     |

#### 날짜 및 시간 타입

| 타입      | 유형         | 용량(Byte) | 범위                                          |

#### TEXT와 BLOB의 차이점

- TEXT: 매우 큰 문자열 데이터를 저장할 때 사용하며, 주로 게시판의 본문 등을 저장할 때 사용합니다.
- BLOB: 이미지, 동영상, 음악 파일 등의 이진 데이터를 저장할 때 사용하며, 주로 프로필 사진, 첨부 파일 등을 저장할 때 사용합니다.
- S3: 대용량 파일을 저장할 때는 클라우드 스토리지 서비스인 Amazon S3 등을 사용하는 것이 좋습니다.

#### ENUM과 SET의 차이점

- ENUM: 여러 개의 값 중 하나를 선택할 때 사용하며, 주로 성별, 학년 등을 저장할 때 사용합니다.
- SET: 여러 개의 값 중 여러 개를 선택할 때 사용하며, 주로 취미, 전공 등을 저장할 때 사용합니다.

## 관계(Relationship)

> 관계는 릴레이션 간의 연관성을 의미합니다.

관계는 릴레이션 간의 연관성을 나타내며, 관계는 다음과 같이 분류됩니다.

### 관계의 종류

#### 1:1 관계(One-to-One Relationship)

1:1 관계는 한 릴레이션의 튜플이 다른 릴레이션의 튜플과 1:1로 대응하는 관계를 의미합니다.

Spring Data JPA에서는 `@OneToOne` 어노테이션을 사용하여 1:1 관계를 설정할 수 있습니다.

예를 들어, 학생과 학번이 1:1 관계입니다. 한 학생은 한 학번을 가지며, 한 학번은 한 학생에게만 부여됩니다.

#### 1:N 관계(One-to-Many Relationship)

1:N 관계는 한 릴레이션의 튜플이 다른 릴레이션의 튜플과 1:N으로 대응하는 관계를 의미합니다.

Spring Data JPA에서는 `@OneToMany` 어노테이션을 사용하여 1:N 관계를 설정할 수 있습니다.

예를 들어, 학과와 학생이 1:N 관계입니다. 한 학과에는 여러 명의 학생이 속할 수 있으며, 한 명의 학생은 한 학과에만 속할 수 있습니다.

#### N:M 관계(Many-to-Many Relationship)

N:M 관계는 한 릴레이션의 튜플이 다른 릴레이션의 튜플과 N:M으로 대응하는 관계를 의미합니다.

Spring Data JPA에서는 `@ManyToMany` 어노테이션을 사용하여 N:M 관계를 설정할 수 있습니다.

예를 들어, 학생과 강의가 N:M 관계입니다. 한 명의 학생은 여러 개의 강의를 수강할 수 있으며, 한 강의에는 여러 명의 학생이 수강할 수 있습니다.

### 관계의 종류

- 단방향 관계: 한 릴레이션에서 다른 릴레이션으로만 관계를 설정하는 것
- 양방향 관계: 두 릴레이션 간에 서로 관계를 설정하는 것
- 연관 관계: 두 릴레이션 간에 연관성을 설정하는 것
- 의존 관계: 한 릴레이션의 속성이 다른 릴레이션의 속성에 의존하는 것

### 관계의 제약 조건

| 제약조건                | 설명                                                                                               | MySQL               |
| ----------------------- | -------------------------------------------------------------------------------------------------- | ------------------- |
| 무결성 제약 조건        | 데이터베이스에 저장된 데이터의 무결성을 보장하기 위한 제약 조건                                    | -                   |
| 참조 무결성 제약 조건   | 외래 키(Foreign Key)가 참조하는 릴레이션의 기본 키(Primary Key) 값이 존재하는지 확인하는 제약 조건 | FOREIGN KEY         |
| 개체 무결성 제약 조건   | 기본 키(Primary Key) 값이 NULL이 아니며, 중복되지 않는지 확인하는 제약 조건                        | PRIMARY KEY, UNIQUE |
| 도메인 무결성 제약 조건 | 속성의 값이 도메인에 속하는지 확인하는 제약 조건                                                   | CHECK               |
| 키 제약 조건            | 기본 키(Primary Key)와 후보 키(Candidate Key)를 설정하는 제약 조건                                 | PRIMARY KEY, UNIQUE |
| NULL 제약 조건          | 속성의 값이 NULL이 아닌지 확인하는 제약 조건                                                       | NOT NULL            |
| 기본 키 제약 조건       | 기본 키(Primary Key)를 설정하는 제약 조건                                                          | PRIMARY KEY         |
| 외래 키 제약 조건       | 외래 키(Foreign Key)를 설정하는 제약 조건                                                          | FOREIGN KEY         |
| 유일성 제약 조건        | 속성의 값이 중복되지 않는지 확인하는 제약 조건                                                     | UNIQUE              |

### 키(Key)

> 키는 릴레이션에서 튜플을 식별하기 위한 속성의 집합을 의미합니다.

키의 유일성(Uniqueness)은 릴레이션에서 튜플을 식별하기 위한 속성의 집합이 서로 중복되지 않는 것을 의미합니다.

키의 최소성(Minimality)은 릴레이션에서 튜플을 식별하기 위한 속성의 집합이 최소한의 속성으로 구성되어 있는 것을 의미합니다.

#### 키의 종류

| 종류                   | 설명                                       |
| ---------------------- | ------------------------------------------ |
| 슈퍼 키(Super Key)     | 유일성을 만족하는 키의 집합                |
| 후보 키(Candidate Key) | 유일성과 최소성을 만족하는 키의 집합       |
| 기본 키(Primary Key)   | 후보 키 중에서 기본 키로 선정된 키         |
| 대체 키(Alternate Key) | 후보 키 중에서 기본 키로 선정되지 않은 키  |
| 자연 키(Natural Key)   | 엔티티가 가지고 있는 속성 중에서 유일한 키 |
| 인조 키(Surrogate Key) | 인위적으로 생성된 키                       |
| 외래 키(Foreign Key)   | 다른 릴레이션의 기본 키를 참조하는 키      |

## ERD(Entity-Relationship Diagram)

> ERD(Entity-Relationship Diagram)는 엔티티(Entity)와 엔티티 간의 관계(Relationship)를 그래픽으로 표현한 것입니다.

ERD는 데이터베이스의 구조를 시각적으로 표현하여, 데이터베이스의 설계 및 구조를 이해하기 쉽게 도와줍니다. ERD Cloud, Lucidchart 등의 도구를 사용하여 ERD를 그릴 수 있습니다.

비정형 데이터베이스에서는 ERD 대신 스키마가 사용됩니다. 스키마는 데이터베이스의 구조를 정의하는 것으로, 각 엔티티의 릴레이션, 속성 등을 정의합니다.

## 정규화(Normalization)

> 정규화(Normalization)는 데이터베이스 설계 시 중복을 최소화하고 데이터 무결성을 보장하기 위한 과정입니다.

데이터 무결성(Integrity)은 데이터베이스의 정확성, 일관성, 유효성을 보장하는 것을 의미합니다.

### 정규형 원칙(Normal Form)

정규화는 정규형(Normal Form)이라는 원칙을 따릅니다. 정규형은 데이터베이스의 테이블이 특정 조건을 만족하는지를 나타내는 기준입니다.

#### 제 1 정규형(1NF)

- 테이블의 모든 속성은 원자값(Atomic Value)을 가져야 합니다.
- 원자값이란 더 이상 분해할 수 없는 값으로, 속성이 다중값을 가지면 안됩니다.
- 예를 들어, 학생 테이블의 주소 속성이 '서울시 강남구'와 '서울시 강북구'로 이루어진다면, 이는 1NF를 만족하지 않습니다.

#### 제 2 정규형(2NF)

- 테이블이 1NF이어야 합니다.
- 테이블의 모든 속성이 기본키에 완전 함수 종속(Fully Functional Dependent)되어야 합니다.
- 완전 함수 종속이란 기본키의 일부가 아닌 전체에 종속되어야 한다는 것을 의미합니다.
- 예를 들어, 학생 테이블의 학번, 이름, 학과 속성이 있을 때, 학번만으로 이름과 학과를 결정할 수 있어야 합니다.

#### 제 3 정규형(3NF)

- 테이블이 2NF이어야 합니다.
- 테이블의 모든 속성이 기본키에 이행적 함수 종속(Transitive Functional Dependent)되어서는 안됩니다.
- 이행적 함수 종속이란 A -> B, B -> C일 때, A -> C가 성립하는 것을 의미합니다.
- 예를 들어, 학생 테이블의 학번, 이름, 학과, 학과장 속성이 있을 때, 학번 -> 학과, 학과 -> 학과장이 성립하면 학번 -> 학과장이 성립하므로 3NF를 만족하지 않습니다.

#### BCNF(Boyce-Codd Normal Form)

- 테이블이 3NF이어야 합니다.
- 모든 결정자가 후보키(Candidate Key)이어야 합니다.
- 결정자(decider)란 함수 종속 관계에서 특정 종속자(dependent)를 결정하는 요소를 의미합니다. `A -> B`에서 `A`가 결정자, `B`가 종속자입니다.
- 예를 들어, 학생 테이블의 학번, 이름, 학과 속성이 있을 때, 학번이 후보키라면 학번 -> 이름, 학번 -> 학과가 성립해야 합니다.

## 트랜잭션(Transaction)

> 트랜잭션(Transaction)은 데이터베이스에서 하나의 논리적 기능을 수행하기 위한 작업의 단위를 의미합니다.

DBMS는 쿼리를 통해 데이터베이스에 접근하므로, 트랜잭션은 여러 개의 쿼리들을 하나로 묶는 단위라고도 할 수 있습니다. 트랜잭션의 특징 4가지를 묶어 통상적으로 ACID라는 약어로 표현됩니다.

### 원자성(Atomicity)

> 트랜잭션은 데이터베이스에 모두 반영되거나 전혀 반영되지 않아야 합니다.

커밋과 롤백을 통해 트랜잭션의 원자성을 보장할 수 있습니다. 커밋은 트랜잭션을 성공적으로 처리했을 때, 롤백은 트랜잭션을 취소하고 이전 상태로 되돌릴 때 사용합니다.

예를 들어, A 계좌에서 B 계좌로 이체하는 트랜잭션에서 A 계좌에서의 출금은 성공했지만, B 계좌로의 입금이 실패하면 트랜잭션 전체가 실패해야 합니다.

#### 커밋(Commit)

커밋은 여러 쿼리가 성공적으로 처리되었다고 확정하는 명령어입니다. 커밋은 트랜잭션 단위로 수행되며, 커밋이 완료되면 트랜잭션은 데이터베이스에 영구적으로 반영됩니다.

#### 롤백(Rollback)

롤백은 트랜잭션을 취소하고 이전 상태로 되돌리는 명령어입니다. 롤백은 트랜잭션 단위로 수행되며, 롤백이 완료되면 트랜잭션은 데이터베이스에 반영되지 않습니다.

#### 전파(Propagation)

전파는 트랜잭션의 상태를 다른 트랜잭션에 전달하는 것을 의미합니다.

SpringBoot에서는 `@Transactional` 어노테이션을 사용하여 트랜잭션을 설정할 수 있습니다.

### 일관성(Consistency)

> 트랜잭션은 허용된 방식으로만 데이터를 변경해야 합니다.

데이터베이스에 저장된 데이터는 트랜잭션을 통해 일관성을 유지해야 합니다. 예를 들어, 계좌 이체 트랜잭션에서 A 계좌에서의 출금은 B 계좌로의 입금과 동시에 이루어져야 합니다.

### 격리성(Isolation)

> 트랜잭션은 다른 트랜잭션에 영향을 받지 않고 독립적으로 수행되어야 합니다.

트랜잭션은 다른 트랜잭션과 격리되어야 하며, 트랜잭션 간의 데이터 변경은 서로에게 영향을 주지 않아야 합니다.

#### 격리 수준(Isolation Level)

격리 수준은 트랜잭션 간의 격리 정도를 나타내며, 다음과 같이 4가지로 나뉩니다.

| 격리 수준(Isolation Level) | 설명                                                     | 동시성 | 격리성 |
| -------------------------- | -------------------------------------------------------- | ------ | ------ |
| READ UNCOMMITTED           | 다른 트랜잭션이 커밋하지 않은 데이터를 읽을 수 있습니다. | 높음   | 낮음   |
| READ COMMITTED             | 다른 트랜잭션이 커밋한 데이터만 읽을 수 있습니다.        | 높음   | 보통   |
| REPEATABLE READ            | 같은 쿼리를 실행해도 항상 같은 결과를 보장합니다.        | 보통   | 보통   |
| SERIALIZABLE               | 트랜잭션 간의 격리를 가장 강하게 보장합니다.             | 낮음   | 높음   |

#### 팬텀 리드(Phantom Read)

팬텀 리드는 같은 쿼리를 실행해도 결과가 달라지는 현상을 의미합니다. 격리 수준이 낮을수록 팬텀 리드가 발생할 가능성이 높아집니다.

#### 더티 리드(Dirty Read)

더티 리드는 다른 트랜잭션이 커밋하지 않은 데이터를 읽는 현상을 의미합니다. 격리 수준이 낮을수록 더티 리드가 발생할 가능성이 높아집니다.

#### 반복 가능하지 않은 읽기(Non-Repeatable Read)

반복 가능하지 않은 읽기는 같은 쿼리를 실행해도 결과가 달라지는 현상을 의미합니다. 격리 수준이 낮을수록 반복 가능하지 않은 읽기가 발생할 가능성이 높아집니다.

### 지속성(Durability)

> 트랜잭션이 성공적으로 완료되면, 해당 결과는 영구적으로 반영되어야 합니다.

데이터베이스는 트랜잭션의 결과를 영구적으로 저장해야 하며, 시스템 장애가 발생해도 데이터의 일관성을 보장해야 합니다.

#### 복구(Recovery)

복구는 데이터베이스의 시스템 장애로 인해 손상된 데이터를 복구하는 과정을 의미합니다.

#### 체크섬(Checksum)

체크섬은 데이터의 무결성을 검사하기 위한 값으로, 데이터의 변경 여부를 확인할 수 있습니다.

#### 저널링(Journaling)

저널링은 데이터베이스의 변경 이력을 기록하는 것으로, 데이터의 변경 이력을 추적할 수 있습니다.
