## 데이터베이스의 종류

> 관계형 데이터베이스와 NoSQL 데이터베이스는 데이터를 저장하고 관리하는 방식에 따라 데이터베이스를 분류한 것입니다.

### 관계형 데이터베이스

관계형 데이터비에스(RDMBS, Relational Database Management System)는 행과 열을 가지는 테이블 형식 데이터를 저장하는 형태의 데이터베이스입니다.

- 각 열(Column)는 데이터 속성을 나타냅니다.
- 각 행(Row)은 데이터의 인스턴스를 나타냅니다.
- 각 테이블은 해당 테이블을 고유하게 식별하는 식별자 열인 기본 키(primary key)를 가집니다.
- 다른 테이블 간의 관계를 나타내기 위해 외래 키(foreign key)를 사용합니다.
- SQL(Structured Query Language)을 사용하여 데이터를 관리합니다. 이에 관계형 데이터베이스를 SQL 데이터베이스라고도 합니다.
- 대표적인 관계형 데이터베이스로는 MySQL, PostgreSQL, Oracle, SQL Server 등이 있습니다.

#### MySQL

MySQL은 대부분의 운영체제와 호환되며 현재 가장 많이 사용하는 데이터베이스입니다.

- C, C++로 작성된 오픈 소스 관계형 데이터베이스 관리 시스템입니다.
- 대용량 데이터베이스를 위해 설계되었으며, 롤백, 커밋, 이중 암호 지원 보안 등의 기능을 제공합니다.
- 모듈식 아키텍쳐로 쉽게 스토리지 엔진을 변경할 수 있습니다.
- 커넥터 API 및 서비스 계층을 통해 다양한 프로그래밍 언어와 통합할 수 있습니다.

#### PostgreSQL

PostgresSQL은 MySQL보다 더 많은 기능을 제공하는 객체 관계형 데이터베이스 관리 시스템입니다.

- 디스크 조각이 차지하는 영역을 회수할 수 있는 장치인 VACUUM을 제공합니다.
- 지정 시간에 복구하는 기능, 로깅, 접근 제어, 중첩된 트랜잭션, 백업 및 복구 기능을 제공합니다.

#### MySQL과 PostgreSQL의 차이점

| 항목                   | MySQL                                           | PostgreSQL                                                                     |
| ---------------------- | ----------------------------------------------- | ------------------------------------------------------------------------------ |
| **데이터베이스 기술**  | 순수 관계형 데이터베이스 관리 시스템            | 객체-관계형 데이터베이스 관리 시스템                                           |
| **데이터 타입**        | 숫자, 문자, 날짜 및 시간, 공간 JSON 데이터 유형 | 기하학, 열거형, 네트워크 주소, 배열, 범위, XML을 포함한 모든 MySQL 데이터 유형 |
| **ACID 지원**          | InnoDB 및 NDB 클러스터 엔진에서만 지원          | 모든 구성에서 완전한 ACID 준수                                                 |
| **동시성 제어 (MVCC)** | InnoDB 엔진에서만 MVCC 지원                     | 모든 구성에서 MVCC 완전 지원                                                   |
| **인덱스 유형**        | B-tree, R-tree 인덱스 지원                      | B-tree, 해시, 표현 인덱스, 부분 인덱스 등 다양한 옵션 지원                     |
| **성능**               | 높은 빈도의 읽기 작업 성능                      | 높은 빈도의 쓰기 작업 성능                                                     |
| **학습 난이도**        | 입문자 친화적, 짧은 학습 곡선                   | 초기 설정 및 문제 해결이 복잡하여 난이도가 높은 편                             |

### NoSQL 데이터베이스

NoSQL(Not only SQL) 데이테베이스는 데이터를 저장하고 관리하기 위해 다양한 데이터 모델을 사용하는 데이터베이스의 총칭입니다.

- 각 NoSQL 데이터베이스는 특정 데이터 모델을 위해 특별히 구축되었습니다.
- 현대의 웹 애플리케이션은 다양한 소스의 데이터를 처리하는데, 이러한 이질적인 데이터는 관계형 데이터베이스로 처리하기 어려운 경우가 많습니다.
- 이에 NoSQL 데이터베이스를 비관계형 데이터베이스라고도 합니다.
- 대표적인 NoSQL 데이터베이스로는 MongoDB, Redis, Cassandra 등이 있습니다.

#### NoSQL 데이터베이스의 유형

1. **키-값 데이터베이스(Key-Value Database)**: 키를 고유한 식별자로 사용하는 키-값 쌍의 컬렉션으로 데이터를 저장합니다.
2. **도큐먼트 데이터베이스(Document Database)**: 애플리케이션 코드에 사용되는 것과 동일한 문서 모델 형식을 사용합니다.
3. **그래프 데이터베이스(Graph Database)**: 노드를 사용하여 데이터 엔터티를 저장하고 엣지로는 엔터티 간의 관계를 저장합니다.
4. **인 메모리 데이터베이스(In-Memory Database)**: 디스크에 액세스 할 필요가 없도록 설계된 데이터베이스로, 데이터를 메모리에 저장합니다.

#### MongoDB

JSON을 통해 데이터에 접근할 수 있고, BSON(Binary JSON) 형식으로 데이터를 저장하는 NoSQL 데이터베이스입니다.

- 와이드타이거 엔진이 기본 스토리지 엔진으로 사용됩니다.
- 키-값 데이터 모델에서 확장된 도큐먼트 기반의 데이터 모델을 사용합니다.
- 확장성이 뛰어나며 빅데이터 처리에 적합합니다.
- 고가용성, 샤딩, 레플리카 셋을 지원합니다.
- 다양한 도메인의 데이터베이스를 분석하거나 로깅 등을 구현할 때 강점을 보입니다.

#### Redis

Redis는 인메모리 데이터베이스로 키-값 데이터 모델을 사용하는 NoSQL 데이터베이스입니다.

- 기본적인 데이터 타입은 문자열(string)이며 최대 512MB까지 저장할 수 있습니다.
- 문자열 외에도 리스트, 셋, 정렬된 셋, 해시, 비트맵, 공간 인덱스 등 다양한 데이터 타입을 지원합니다.
- pub/sub 메시징 시스템을 지원하며, 메모리에 데이터를 저장하므로 빠른 읽기/쓰기 속도를 제공합니다.

## 인덱스

> 인덱스는 쿼리 패턴에 대한 성능을 향상시키기 위해 데이터베이스 테이블의 열에 대한 추가적인 데이터 구조를 생성하는 방법입니다.

테이블에서 쓰기가 이루어질 때마다 테이블의 모든 인덱스가 업데이트 되어야 하므로 테이블이 크고 쓰기 작업이 많은 환경에서는 시스템 리소스가 많이 사용될 수 있습니다.

### B+트리 인덱스
B+트리는 데이터베이스의 인덱싱을 위해 최적화된 자료구조로, 각 노드가 여러 개의 키를 가질 수 있는 균형 트리입니다.

- 루트 노드 : 트리의 최상단에 위치하는 노드입니다. 단 하나만 존재하며, 다른 브랜치 노드와 동일한 역할을 수행합니다.
- 브랜치 노드 : 루트와 리프 사이에 위치하는 노드로, 자식 노드의 범위를 구분하는 키값들을 가지고 있습니다. 각 키값은 "이 값보다 큰 데이터는 다음 포인터로 가라"는 의미를 가집니다.
- 리프 노드 : 트리의 최하단에 위치하며 실제 모든 데이터가 저장됩니다. 리프 노드들은 연결 리스트 형태로 연결되어 있어 순차 검색이 가능합니다.

#### B+트리의 생성
1. 리프 노드는 최대 차수까지 데이터를 순차적으로 저장합니다.
2. 리프 노드가 가득 차면 중간값을 브랜치 노드로 복사하고 리프 노드를 분할합니다.
   * 이때 브랜치 노드로 복사된 값도 리프 노드에 남아있습니다.
3. 브랜치 노드가 가득 차면 마찬가지로 중간값을 상위 브랜치 노드로 복사하고 분할합니다.
4. 이러한 과정이 반복되며 트리가 성장합니다.

#### B+트리의 탐색 과정
1. 루트 노드에서 시작해 브랜치 노드를 따라 내려갑니다.
2. 각 브랜치 노드에서는 다음과 같은 규칙으로 이동합니다:
   * 찾는 값이 현재 키값보다 작으면 왼쪽으로 이동
   * 찾는 값이 현재 키값 이상이면 오른쪽으로 이동
3. 리프 노드에 도달하면 실제 데이터를 순차 검색합니다.
4. 리프 노드에서 찾고자 하는 값이 있으면 반환하고, 없으면 검색 실패를 반환합니다.

### 인덱스 만드는 방법

MySQL은 클러스터형 인덱스와 세컨더리 인덱스가 있습니다.

#### 클러스터형 인덱스

클러스터형 인덱스는 테이블의 기본 키에 대한 인덱스를 의미합니다.

- primary key 옵션을 사용하여 기본 키를 지정하면 클러스터형 인덱스가 생성됩니다.
- unique not null 옵션을 사용하여 클러스터형 인덱스를 생성할 수도 있습니다.
- 인덱스와 데이터가 같은 B-Tree 구조에 저장되어 있으므로 검색이 빠릅니다.
- 한 테이블에는 하나의 클러스터형 인덱스만 생성할 수 있습니다.

#### 세컨더리 인덱스

세컨더리 인덱스는 테이블의 기본 키가 아닌 열에 대한 인덱스를 의미합니다.

- create index 문을 사용하여 세컨더리 인덱스를 생성할 수 있습니다.
- 세컨더리 인덱스는 B-Tree 구조를 따르지만 실제 데이터는 포함하지 않고, 클러스터형 인덱스를 참조하는 포인터를 저장합니다.
- 여러 개의 세컨더리 인덱스를 생성할 수 있습니다.
- 특정 열에 대한 세컨더리 인덱스를 생성하면 해당 열에 대한 검색이 빨라집니다.

#### MongoDB 인덱스

도큐먼트를 만들면 자동으로 ObjectID가 형성되며, 해당 키가 기본 키로 설정됩니다.

- createIndex() 메서드를 사용하여 인덱스를 생성할 수 있습니다.
- 기본 키와 세컨더리 키를 같이 쓰는 복합 인덱스를 생성할 수 있습니다.

### 인덱스 최적화 기법

1. 인덱스는 비용이다 :

   - 인덱스 리스트를 탐색하고 그 다음으로 데이터를 탐색하기에 탐색 비용이 발생합니다.
   - 데이터를 수정할 때 인덱스도 수정해야 하므로 쓰기 비용이 발생합니다.
   - 모든 필드에 인덱스를 설정할 수 없고, 데이터베이스의 크기가 커질수록 인덱스의 크기도 커지므로 메모리 사용량이 증가합니다.

2. 테스팅 :

- 인덱스를 생성하기 전에 쿼리를 테스트하여 어떤 인덱스가 필요한지 확인합니다.
- MySQL에서는 EXPLAIN을 사용하여 쿼리의 실행 계획을 확인할 수 있습니다.
- MongoDB에서는 explain() 메서드를 사용하여 쿼리의 실행 계획을 확인할 수 있습니다.

3. 복합 인덱스는 같음, 정렬, 다중 값, 카디널리티 순으로 생성하라 :
   - 여러 필드를 기반으로 조회할 때 복합 인덱스르 생성하는데, 이 순서에 따라 인덱스의 성능이 달라집니다.
   - 같음을 비교하는 `==`이나 `equal` 연산자를 사용하는 필드를 첫 번째로 사용합니다.
   - 정렬이 필요한 필드를 두 번째로 사용합니다.
   - 쿼리가 많은 값을 출력하는 필드를 세 번째로 사용합니다.
   - 유니크한 정도인 카디널리티가 높은 필드를 마지막으로 사용합니다.

## 조인

> 조인은 두 개 이상의 테이블을 연결하여 하나의 결과 집합으로 반환하는 연산입니다.

NoSQL의 경우 조인의 성능이 나쁘기 때문에 지원하지 않는 경우도 있으며, 조인이 많이 필요한 데이터 유형의 경우 관계형 데이터베이스를 사용하는 것이 좋습니다.

### 조인의 종류

| 조인 유형                     | 설명                                                              | SQL 예시                                                                   |
| ----------------------------- | ----------------------------------------------------------------- | -------------------------------------------------------------------------- |
| 내부 조인(INNER JOIN)         | 두 테이블 간의 일치하는 행을 반환합니다.                          | `SELECT * FROM table1 INNER JOIN table2 ON table1.column = table2.column;` |
| 왼쪽 조인(LEFT OUTER JOIN)    | 왼쪽 테이블의 모든 행과 오른쪽 테이블의 일치하는 행을 반환합니다. | `SELECT * FROM table1 LEFT JOIN table2 ON table1.column = table2.column;`  |
| 오른쪽 조인(RIGHT OUTER JOIN) | 오른쪽 테이블의 모든 행과 왼쪽 테이블의 일치하는 행을 반환합니다. | `SELECT * FROM table1 RIGHT JOIN table2 ON table1.column = table2.column;` |
| 합집합 조인(FULL OUTER JOIN)  | 두 테이블의 모든 행을 반환합니다.                                 | `SELECT * FROM table1 FULL JOIN table2 ON table1.column = table2.column;`  |
| 교집합 조인(CROSS JOIN)       | 두 테이블의 모든 행을 반환합니다.                                 | `SELECT * FROM table1 CROSS JOIN table2;`                                  |

### 데이터베이스 조인의 원리

데이터베이스의 조인은 중첩 루프 조인, 정렬 병합 조인, 해시 조인 등의 원리를 기반으로 동작합니다.

다양한 조인 방식의 특징과 원리를 정리한 표입니다. 각 조인 방식은 데이터의 양, 인덱스 여부, 테이블 크기에 따라 장단점이 다르므로, 적절한 상황에서 사용해야 합니다.

| 조인 방식      | 개념 및 원리                                                 | 장점                                 | 단점                                     | 사용 사례 및 특징                          |
| -------------- | ------------------------------------------------------------ | ------------------------------------ | ---------------------------------------- | ------------------------------------------ |
| 중첩 루프 조인 | 두 테이블에서 외부 루프의 각 행을 내부 루프의 모든 행과 비교 | 구현이 간단하며, 작은 테이블에 적합  | 테이블이 크면 성능 저하                  | 인덱스가 잘 설정된 작은 테이블 조인        |
| 정렬 병합 조인 | 두 테이블을 정렬한 후 병합하여 매칭하는 방식                 | 정렬된 데이터에서는 빠른 성능을 보임 | 정렬 과정이 추가되어 비용 발생 가능      | 대용량 데이터, 정렬된 상태의 테이블에 적합 |
| 해시 조인      | 작은 테이블로 해시 테이블을 생성하고 큰 테이블과 비교        | 대규모 데이터에서 빠른 조인 가능     | 메모리 사용량이 크며 해시 생성 비용 있음 | 대규모 테이블에서 빠르게 조인할 때 사용    |

#### MySQL의 해시 조인

MySQL 8.0부터 해시 조인이 추가되어, 해시 조인을 사용하면 중첩 루프 조인과 정렬 병합 조인의 단점을 보완할 수 있습니다.

1. 빌드 단계

   - 입력 테이블 중 하나를 기반으로 메모리 내 해시 테이블을 빌드합니다.
   - 조인에 사용되는 필드 중 하나가 해시 테이블의 해시 키가 됩니다.

2. 프로브 단계
   - 다른 입력 테이블을 스캔하면서 해시 테이블과 일치하는 행을 찾습니다.
   - 해시 테이블의 해시 키를 사용하여 일치하는 행을 찾으므로 빠른 성능을 보장합니다.
